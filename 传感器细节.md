# PN532 NFC 模块 I2C 联调注意事项

> STM32F103C8T6 + PN532（I2C 模式, PB6-SCL / PB7-SDA）

## 1. 硬件接线

| PN532 | STM32F103 |
|-------|-----------|
| VCC   | 3.3V      |
| GND   | GND       |
| SCL   | PB6       |
| SDA   | PB7       |
| IRQ   | 不接       |
| RST   | 不接       |

## 2. I2C 配置

- **速率**: 100kHz（标准模式），高频可能不稳定
- **地址**: PN532 默认 7-bit 地址 `0x24`，HAL 需左移 1 位 → `0x48`
- CubeMX 配置: `I2C1` → Standard mode → PB6/PB7

## 3. STM32F103 I2C 硬件硅缺陷（关键！）

**现象**: `HAL_I2C_Master_Receive()` 读 N 字节时，**最后一字节可能被损坏**（变成 `0xFF` 或其他值）。

**解决方案**:
- 读 N 字节时，**实际读 N+1 字节**，只用前 N 个
- 例: ACK 是 7 字节，读 8 或 9 字节，`memcmp` 只比前 7
- 例: 响应帧最大 30 字节，读 41 字节

## 4. 不要单独读 status 字节（关键！）

**错误做法**（消费数据流程破坏者）:
```c
// 轮询 status=0x01
uint8_t s;
read_i2c(&s, 1, 10);  // 读 1 字节
if (s == 0x01) { ... }
```
这样读完 1 字节后，后续 `read_i2c` 读到的数据偏移了 1 字节，导致帧解析错误。

**正确做法**:
- **ACK 阶段**: `HAL_Delay(2~5ms)` 后直接读 status+ACK（共 8~9 字节），跳过 status 字节只验 preamble
- **响应阶段**: `HAL_Delay(2ms)` 后直接读 status+帧（共 41 字节），从 `buf[1]` 开始解析帧

```
发帧  →  HAL_Delay(2ms)  →  read_i2c(9)  →  校验 ack[1..3] preamble
发帧  →  HAL_Delay(2ms)  →  read_i2c(41) →  从 buf[1] 解析帧
```

## 5. PN532 帧协议

### 发送帧（MCU → PN532）
```
00 00 FF  LEN  ~LEN  D4  [payload...]  CHKSUM  00
```

- `LEN` = TFI(1) + payload 长度
- `~LEN` = 0x100 - LEN
- `CHKSUM` = 0x100 - (D4 + payload 各字节累加) 的低 8 位

### ACK 响应（PN532 → MCU）
```
01  00 00 FF 00 FF 00
```
7 字节: `01` 是 status, 后面 6 字节是 ACK 帧本体

### 数据响应（PN532 → MCU）
```
01  00 00 FF  LEN  ~LEN  D5  [payload...]  CHKSUM  00
```
- `D5` = D4+1（响应方向）
- payload 第 1 字节是命令码 + 1

### Payload 长度计算（易错点）
```
LEN = TFI(1) + payload
payload_字节数 = LEN - 1     ← 不是 LEN - 2
```

例: `D5 03 32 01 06 07 E8`  
LEN=0x06 → payload=5 字节 → `D5 03 32 01 06`（TFI+数据）

## 6. 调试日志参考

### 正常通信
```
[PN532] Init...
[PN532] I2C scan...
[PN532] found 0x48
[PN532] GetVersion...
[PN532] ACK: 01 00 00 FF 00 FF 00 xx xx    ← preamble 匹配即可
[PN532] resp: 01 00 00 FF 06 FA D5 03 32 01 06 07 E8 ...
[PN532] FW IC=32 Ver=1.6 Sup=07
[PN532] SAM...
[PN532] SAM OK
[PN532] resp: ... D5 4B 01 01 00 04 08 04 53 16 FF 06 60 00
```

### 扫描到卡
- `D5 4B` = InListPassiveTarget 响应
- 第 3 字节表示卡片数量（1=1 张卡）
- 后续是目标数据（TAG+SENS_RES+SEL_RES+UID）

## 7. 完整读取流程

```
1. 发帧:       write_i2c(frame, frame_len)
2. 短延时:     HAL_Delay(2)
3. 读 ACK:     read_i2c(buf, 9, 50)    // F103 bug: 读 N+1
4. 校验 ACK:   buf[0]=01, buf[1..3]=00 00 FF  → 宽松通过
5. 短延时:     HAL_Delay(2)
6. 读响应:     read_i2c(buf, 41, 200)   // F103 bug: 读 N+1
7. 解析帧:     从 buf[1] 开始（跳过 status）
               LEN=buf[4], payload=buf[6..6+LEN], payload_len = LEN-1
               checksum 校验: out[LEN-1] = 计算的 checksum
               postamble: out[LEN] == 0x00
```

## 8. I2C 地址扫描

PN532 的可能地址:
```
0x24 << 1  = 0x48  ← 默认
0x25 << 1  = 0x4A  ← SEL0 拉高
0x26 << 1  = 0x4C  ← SEL1 拉高
```

用 `HAL_I2C_IsDeviceReady()` 依次扫描确认。
